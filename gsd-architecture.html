0 /gsd:new-project — Dream Extraction

User's idea → PROJECT.md + REQUIREMENTS.md + ROADMAP.md

What happens: The orchestrator asks open-ended questions following the "dream extraction" philosophy from references/questioning.md. This is the BEST part of GSD — genuinely adaptive, follows energy, challenges vagueness.


Flow:

1. gsd-tools init new-project → detects brownfield, checks existing state

2. Adaptive questioning (no fixed count — loops until "Ready?")

3. Writes PROJECT.md from answers

4. Spawns 4× gsd-project-researcher in parallel (domain, stack, features, pitfalls)

5. gsd-research-synthesizer merges results

6. Generates REQUIREMENTS.md (table stakes + features)

7. gsd-roadmapper creates ROADMAP.md with phase breakdown


Outputs: PROJECT.md REQUIREMENTS.md ROADMAP.md STATE.md config.json


Good Questioning is genuinely adaptive here — no fixed script, follows the thread

Problem ROADMAP phases are created from AI's interpretation of the project, not detailed user input per phase. This is where generic assumptions enter the system.

↓

1 /gsd:discuss-phase N — Context Gathering

Clarify HOW to implement phase N → CONTEXT.md

What actually happens (the problem is here):


1. gsd-tools init phase-op N → loads config, finds phase dir

2. load_prior_context — reads PROJECT.md, REQUIREMENTS.md, STATE.md, all prior CONTEXT.md files

3. scout_codebase — quick grep of existing code for relevant patterns

4. analyze_phase — reads the ROADMAP.md phase description and identifies "gray areas"


⚠ THE CORE ISSUE: Gray areas are generated from one-line ROADMAP descriptions + generic project context. The workflow says "Generate phase-specific gray areas" but in practice:

• The ROADMAP description is typically 1-2 sentences (written by gsd-roadmapper in step 0)
• The "prior context" includes PROJECT.md which describes the whole project generically
• The gray area generator sees "trading app" → generates questions about trade execution, even if this phase is just UI
• The workflow pre-generates 3-4 categories, then asks 4 questions per category
• Total: ~12-16 questions, pre-determined in structure, from AI-generated understanding

5. present_gray_areas — shows 3-4 areas, user picks which to discuss

6. discuss_areas — 4 questions per area via AskUserQuestion (or batch mode)

7. Writes CONTEXT.md with decisions (locked / discretion / deferred)


Problem 1 Questions come from AI's gray area analysis, not from real unknowns the user needs to resolve

Problem 2 Fixed structure: always 3-4 areas × 4 questions. A complex phase gets same depth as a simple one

Problem 3 PROJECT.md (full project scope) bleeds into phase-specific questions — e.g., asking about core trading logic when phase is just "add charts UI"

Problem 4 No back-and-forth — each question is isolated AskUserQuestion, not a conversation thread

Good The prompt DOES say "follow the thread" and "be a thinking partner not interviewer" — but the rigid AskUserQuestion structure fights this

Good Scout codebase step helps ground questions in reality

Good Prior decisions are loaded and not re-asked

↓

2 /gsd:plan-phase N — Plan Creation

CONTEXT.md → RESEARCH.md → PLAN.md files

This is a 3-agent pipeline orchestrated by plan-phase.md workflow:


Step A: Research (if enabled)

Spawns gsd-phase-researcher in fresh 200K context

Reads: CONTEXT.md (locked decisions constrain research scope)

Outputs: RESEARCH.md — standard stack, patterns, pitfalls, code examples


Step B: Plan

Spawns gsd-planner in fresh 200K context

Reads: CONTEXT.md + RESEARCH.md + PROJECT.md + ROADMAP.md + prior SUMMARYs

Creates: PLAN.md files with tasks, waves, dependencies, must-haves


Step C: Verify (loop up to 3×)

Spawns gsd-plan-checker → validates structure, coverage, dependencies

If issues: spawns gsd-planner again with feedback (revision mode)


Problem Planner receives CONTEXT.md which was built from generic questions → plans inherit that genericness

Problem Each agent gets fresh context but only reads what's in the files — no conversational understanding of WHY the user made certain decisions

Good Plan checker loop catches structural issues

Good Goal-backward methodology is solid — derives from outcomes not tasks

Good Vertical slices preferred over horizontal layers

↓

3 /gsd:execute-phase N — Execution

PLAN.md files → code commits → SUMMARY.md files

Wave-based parallel execution:


1. gsd-tools init execute-phase N → loads plans, config, models

2. gsd-tools phase-plan-index N → groups plans into waves by dependencies

3. For each wave (sequential):

   Within wave (parallel if enabled):

   Spawn gsd-executor per plan (fresh 200K each)

   Each executor: reads PLAN.md → executes tasks → commits → writes SUMMARY.md

   STATE.md updated with file locking (O_EXCL)

4. Post-execution: spawn gsd-verifier → quality gate


Good Wave parallelism is well-designed

Good Each executor gets clean context with just its plan

Problem If CONTEXT.md had wrong assumptions, executor builds wrong thing — no feedback loop until verification

↓

4 /gsd:verify-work N — Verification

Check execution results → VERIFICATION.md → optional gap closure

Runs tests, checks artifacts, validates must-haves from PLAN.md

If gaps found → creates gap closure plans → re-execute


Good Multi-layer verification (plan-checker during planning + verifier after execution)

Problem Verification catches structural issues but can't catch "we built the wrong thing" — that's a CONTEXT.md problem

The Information Funnel Problem
The pipeline narrows context at each stage, but the narrowing happens at the wrong point:

Step 0 (new-project): Rich adaptive questioning → PROJECT.md (good, captures vision)
ROADMAP creation: AI interprets PROJECT.md → generates phase descriptions (1-2 sentences each) ← FIRST LOSSY COMPRESSION
Step 1 (discuss-phase): AI reads 1-2 sentence description → generates gray areas → asks preset questions ← SECOND LOSSY COMPRESSION
Step 2 (plan-phase): Planner reads CONTEXT.md (answers to generic questions) → creates specific tasks ← BUILDS ON COMPRESSED FOUNDATION

Result: By the time the executor reads its PLAN.md, it's working from decisions that were answers to questions that were generated from assumptions that were derived from a 1-sentence roadmap entry. Each step adds assumptions, and none of them came from deep discussion with the user about THIS specific phase.
Root Cause Analysis

Problem 1: Generic Gray Areas from Generic Descriptions

In discuss-phase.md, line ~69-106 (gray_area_identification):


The workflow says: "Read the phase goal from ROADMAP.md" — but ROADMAP goals are typically one sentence, written by the AI roadmapper during new-project.


Then it says: "Generate phase-specific gray areas — Not generic categories" — but gives examples that ARE generic patterns:

Phase: "User authentication" → Session handling, Error responses, Multi-device policy, Recovery flow


These ARE the right questions for a greenfield auth phase. But for Phase 8 of a mature project where you just need to add a chart widget? The system still generates fundamental architecture questions because it loaded PROJECT.md which says "trading platform" and it identifies "trade execution" as a gray area.

What happensWhat should happen
AI reads "Phase 8: Charts UI" + PROJECT.md("trading platform") → generates gray areas about trading logic, data refresh rates, error handling for market data AI should see: prior phases solved trading logic, this phase is ONLY about rendering. Gray areas: chart library choice, responsive layout, interaction patterns
3-4 gray areas pre-generated, each gets exactly 4 questions Start with "What's unclear about this phase?" — let the user drive, follow energy like new-project does

Problem 2: AskUserQuestion Kills Conversation Flow

In discuss-phase.md, line ~537-542 (discuss_areas):


Ask 4 questions using AskUserQuestion with header (max 12 chars), 2-3 concrete choices


Compare with new-project's questioning.md: "Start open. Let them dump their mental model. Don't interrupt with structure.", "Follow energy. Whatever they emphasized, dig into that."


The questioning philosophy is great. But discuss-phase ignores it and uses rigid AskUserQuestion menus for everything. This forces multiple-choice answers where freeform discussion would reveal more.


The workflow does have "If 'Other' → ask as plain text" but the user has to actively fight the structure to get there.

Current: Interview modeBetter: Conversation mode
"How should posts be displayed?" → [Cards] [List] [Timeline] [Other] "Tell me about how you envision this phase. What's the most important thing to get right?"
User picks "Cards" → next question. No exploration of WHY cards, what kind, what inspiration. "Cards — interesting. Like Pinterest-style masonry? Or more like Twitter's compact cards? Any apps you'd point to as reference?"

Problem 3: Each Phase Discussed in Isolation

In discuss-phase.md, the workflow is designed for ONE phase at a time:


/gsd:discuss-phase 1 → new context, asks questions, writes CONTEXT.md
/clear
/gsd:discuss-phase 2 → new context, asks questions, writes CONTEXT.md


The workflow DOES load prior CONTEXT.md files (load_prior_context step), but:

• It only reads decisions from prior phases, not the reasoning or discussion

• The DISCUSSION-LOG.md is explicitly marked "Audit trail only. Do NOT use as input to planning, research, or execution agents."

• Related phases (e.g., 5 UI phases) can't share a conversation where one insight informs the next


Your example: If phase 1 discussion reveals "we want a design system with specific tokens," phases 2-5 should inherit that context naturally — not re-discover it from a one-line decision in CONTEXT.md.

Problem 4: The "Questioning Philosophy" Only Lives in new-project

Two completely different questioning approaches:


references/questioning.md (used by new-project only):

• "Dream extraction, not requirements gathering"

• "Follow energy. Whatever they emphasized, dig into that"

• "Challenge vagueness. Never accept fuzzy answers"

• "Know when to stop" (adaptive length)

• Loop: questions → more questions → until user says "ready"


discuss-phase.md (used for every phase):

• AI generates 3-4 gray areas

• 4 questions per area via AskUserQuestion

• "More questions or next area?" gate

• Fixed menu-driven structure


The irony: The MOST leveraged questioning (per-phase, where specifics matter) uses the LEAST adaptive approach. The project-level questioning (where generality is fine) uses the MOST adaptive approach.

Problem 5: Research Agent Reinforces AI Assumptions

In gsd-phase-researcher.md:


The researcher reads CONTEXT.md (which came from generic questions) and researches based on those decisions. If the questions missed the real complexity, the research misses it too.


Example scenario — "Phase 8: Charts UI" for trading app:

1. discuss-phase asks about chart library, colors, responsive layout (generic UI questions)

2. User answers: "use TradingView widget, dark theme, mobile responsive"

3. Researcher researches TradingView widget API, dark theme CSS, responsive patterns

4. Planner creates tasks: "integrate TradingView", "add dark theme", "make responsive"


What was missed: The REAL complexity was websocket data streaming to the chart, handling reconnections, buffering updates during network gaps, syncing chart state with order book. Nobody asked because the gray areas were about UI appearance, not data flow.


A genuine back-and-forth discussion would have surfaced this: "Tell me about the charts — what data feeds them?" → "Oh, it's real-time websocket data" → "That changes everything. Let's talk about the data pipeline first."

Agent Ecosystem — Who Does What
Key insight: Agents never call each other. Workflows (markdown orchestrators) spawn agents and collect results. Each agent gets a fresh context window and communicates only through files (.planning/*.md).

gsd-executor

Executes PLAN.md tasks, creates commits, writes SUMMARY.md
Tools: Read, Write, Edit, Bash, Grep, Glob
Spawned by: execute-phase (parallel, per plan)

gsd-planner

Creates PLAN.md files with task breakdown, waves, dependencies
Tools: Read, Write, Bash, Glob, Grep, WebFetch, Context7
Spawned by: plan-phase (can be called multiple times in revision loop)

gsd-phase-researcher

Tools: Read, Write, Bash, Grep, Glob, WebSearch, WebFetch, Context7

Agent Interaction Pattern
No agent-to-agent communication. All coordination happens through the workflow orchestrator + filesystem:

Workflow reads state → spawns Agent A → Agent A writes file → Workflow reads result → spawns Agent B → Agent B reads Agent A's file

This is fundamentally message passing via filesystem. The "messages" are markdown files. The "mailboxes" are .planning/phases/ directories. The "router" is the workflow markdown.

Proposed Improvements

1. Apply questioning.md Philosophy to discuss-phase

The questioning approach that works great in new-project should be the DEFAULT for discuss-phase too, not just an afterthought buried in "Other" options.

Current: Menu-Driven Interview

  • AI pre-generates 3-4 gray areas from roadmap description
  • User picks which areas to discuss
  • 4 × AskUserQuestion per area (multiple choice)
  • "More or next?" gate after each area
  • CONTEXT.md = answers to AI's questions

Proposed: Conversation-First

  • Start with: "Tell me about this phase. What matters most?"
  • Follow the user's energy and emphasis
  • Use AskUserQuestion ONLY for specific choice points
  • Probe vague answers: "What does 'simple' mean here?"
  • CONTEXT.md = distilled understanding from real conversation
  • Adaptive depth: complex phases get deep discussion, simple phases get quick confirmation

2. Cross-Phase Discussion for Related Phases

5 UI phases discussed together share context, avoid redundant questions, and produce coherent decisions. One insight in phase 2 discussion naturally informs phase 5.
# New command: discuss related phases in single context /gsd:discuss-phases 2,3,5,8 # or: --tag ui # Flow: 1. Load ALL phase descriptions from ROADMAP 2. Identify shared concerns (all UI → shared design system, routing, etc.) 3. Start conversation: "These 4 phases are all UI work. Let's talk about the shared decisions first, then get specific per phase." 4. Shared questions asked ONCE ("design system?", "routing?", "state mgmt?") 5. Phase-specific questions only where phases diverge 6. Adaptive depth: Phase 2 (complex dashboard) gets 15 questions, Phase 8 (simple about page) gets 2 7. Write individual CONTEXT.md files WITH cross-references

3. Codebase-Aware Question Generation

The scout_codebase step already exists but its findings don't strongly shape the questions. If existing code solves 80% of the phase, questions should focus on the remaining 20%.

Current: Scout is Cosmetic

  • Scout greps for related files
  • Findings added as annotations on pre-generated gray areas
  • "Card component exists" shown as a note, doesn't change the question
  • Still asks "how should posts be displayed?" even if pattern is established

Proposed: Scout Drives Questions

  • Scout identifies: what ALREADY EXISTS vs what's NEW
  • Only generate gray areas for NEW decisions
  • "Your app uses Card everywhere — I'll use that. Let me ask about the parts that DON'T have patterns yet."
  • Phase-specific complexity assessment: "This phase touches 3 new files" vs "This phase integrates with 12 existing modules"

4. Richer CONTEXT.md with Reasoning

Currently CONTEXT.md captures WHAT was decided but not WHY or the user's emphasis. Downstream agents can't distinguish "user specifically insisted on X" from "user casually picked X from a menu."
# Current CONTEXT.md decisions section: ### Layout - Card-based layout - Dark theme # Proposed: Add signal strength + reasoning ### Layout - Card-based layout [STRONG — user referenced Pinterest specifically, wants masonry grid with variable heights] - Dark theme [WEAK — user said "probably dark" without strong preference, could go either way if implementation is simpler with light] # This lets the planner and executor understand: # - Card layout: non-negotiable, user has specific vision # - Dark theme: flexible, don't over-invest

5. Lightweight Manifest for Context Loading

Instead of agents loading all files equally, a manifest tells them what's important, what's related, and what to skip. This is the "structured metadata" approach that research showed is the real win.
# .planning/manifest.yml — always loaded first by gsd-tools phases: 08-charts-ui: status: in_progress complexity: medium depends_on: [01-core, 03-data-layer] related_to: [05-dashboard, 09-alerts] key_decisions: - "TradingView widget" [STRONG] - "Dark theme" [WEAK] what_exists: "Card component, useInfiniteQuery, websocket service" what_is_new: "Chart rendering, real-time data binding" focus_areas: ["websocket integration", "chart responsiveness"] skip_areas: ["trade execution", "auth", "core data model"] # Now gsd-tools init can tell agents: # "Focus on websocket integration and chart responsiveness. # Don't load or ask about: trade execution, auth, core data model."
Impact Assessment
Highest impact, lowest effort:

1. Conversation-first discuss-phase — Change the workflow prompt. No code changes needed. Just apply questioning.md philosophy.

2. Signal strength in CONTEXT.md — Add [STRONG/WEAK] annotations. Minimal prompt change, big downstream benefit.

3. Scout-driven questions — Make scout_codebase output filter gray areas instead of just annotating them.

Medium impact, medium effort:

4. Cross-phase discussion — New command + workflow. Requires new prompt but no code changes.

5. manifest.yml — Requires gsd-tools changes to generate and consume it.
