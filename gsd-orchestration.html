Key difference: Uses Skill() not Task() to avoid nested agent freezes

Skill() vs Task()
Why not Task():
Nested Task() spawning causes runtime freezes (issue #686). discuss → plan → execute would be 3 levels deep.
Skill() solution:
Each Skill() runs at the SAME nesting level. The chain stays flat: discuss finishes → plan starts → execute starts. No deep nesting.
Tradeoff:
Skill() runs in the orchestrator's context window, not a fresh one. For autonomous mode this is OK because the orchestrator stays lean (delegates to agents inside each skill).
Could We Consolidate to a Single Orchestrator?
Analysis of whether all workflows should merge into one orchestrator

Current: 8+ Separate Workflow Orchestrators

Each workflow is a separate markdown file with its own orchestration logic:
• new-project.md, discuss-phase.md, plan-phase.md, execute-phase.md, quick.md, autonomous.md, ui-phase.md, map-codebase.md
• Each has duplicated patterns: init → validate → spawn agents → collect results → update state → commit
• Each loads context slightly differently but from the same files
• Autonomous.md already IS a meta-orchestrator (chains discuss → plan → execute)

Arguments FOR Single Orchestrator

  • → Eliminates duplicated init/validate/commit patterns
  • → Single place to understand the full pipeline
  • → Could maintain conversation context across discuss → plan (your insight about related phases)
  • → Autonomous already proves the pattern works
  • → Easier to add cross-phase parallelism
  • → One prompt to maintain, not 8

Arguments AGAINST Single Orchestrator

  • → One massive prompt = slower, more expensive per invocation
  • → Users often want just ONE step (just discuss, just execute)
  • → Context budget: single orchestrator for discuss+plan+execute would eat context
  • → Separation allows independent versioning/testing
  • → Different workflows need different tools (discuss needs AskUserQuestion, execute needs Task)
  • → Breaking change for all existing users
Recommendation: Hybrid — Keep Separate Commands, Add Thin Meta-Layer

Don't merge the workflows. Instead:

1. Keep individual commands (/gsd:discuss-phase, /gsd:plan-phase, etc.) for when users want one step

2. Add a meta-orchestrator that can chain them intelligently:
• /gsd:discuss-phases 2,3,5 — discuss related phases in one context
• /gsd:run 2 — full pipeline: discuss → plan → execute in one command
• /gsd:run 2,3 --parallel — parallel phases with dependency awareness

3. Share the improvements across workflows via shared references:
• references/conversation-first.md — the questioning philosophy, used by both new-project AND discuss-phase
• references/context-loading.md — shared manifest-aware loading

This gives you consolidation benefits (shared context, cross-phase awareness) without the downsides (massive single prompt, breaking changes).

What the Agents Actually Need (Simplification Opportunity)

Looking at the agent matrix, many agents are very similar:

Could merge:
• gsd-phase-researcher + gsd-ui-researcher → single researcher with focus parameter
• gsd-plan-checker + gsd-ui-checker + gsd-integration-checker → single checker with dimension parameter
• gsd-verifier + gsd-nyquist-auditor → single verifier with depth parameter

This would reduce 15 agents → ~8 agents while maintaining the same capabilities.
The difference between them is mostly the prompt, not the capability.
